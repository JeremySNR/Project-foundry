# AGENTS.md — fast orientation for AI agents working on this repo

> **Maintenance rule (non-negotiable):** any PR that adds, removes, or changes a
> feature MUST update this file in the same PR (and the README where user-facing).
> This document is the fast path for the next agent; if it drifts from the code,
> it is worse than useless. Treat it like a test: stale doc = failing build.

## What this project is

Foundry is a **governance control plane for AI coding agents**: a ticket goes in
(Linear / GitHub Issues / Jira), Foundry analyzes readiness, routes it to a repo,
classifies risk, produces a plan, gets a **human approval**, dispatches a coding
agent (Cursor, Claude Code, signed webhook, manual), then watches the PR/MR
(GitHub / GitLab) and re-dispatches the agent on CI failures or review requests —
policy-gated, capped, budgeted, and fully audited. It deliberately **stops at a
reviewed PR**: no auto-merge, no deploys, no migrations. The safety gates are the
product, not overhead. Canonical product statement: `VISION.md`.

**Looking for work, or about to start some?** The backlog lives in
[GitHub Issues](https://github.com/JeremySNR/Project-foundry/issues): bugs and
hardening are labelled `bug`, the strategic roadmap items `enhancement`. Assign
yourself (or comment) before starting, work one issue — or one clearly-scoped
slice of one — per PR, and split slices into sub-issues rather than new
top-level items. New top-level roadmap items need a human decision, not an
agent's.

## The run lifecycle (what the orchestrator does)

```
webhook intake → analyze (ready? AC present?) → enrich (which repo?) →
classify risk → plan → POLICY GATE → human approval → dispatch agent →
PR opens → diff-aware risk re-check on every push → CI fail / changes
requested → policy-gated retry (capped, budgeted) → merged/closed →
outcome recorded into delivery memory
```

Terminal states: merged, blocked (never retried), rejected, failed,
needs-clarification (re-triggerable — one *active* run per issue, not one ever).

## Honest map of the intelligence layers

Know what is smart and what is deliberately dumb before you "improve" anything:

| Layer | Mechanism | Where |
| --- | --- | --- |
| Ticket readiness / AC extraction | Regex + structure heuristics (default) **or** OpenAI structured output (`analyzer.provider: openai`). LLM failures degrade to the heuristic analyzer, recorded in the analysis `assumptions` — an OpenAI outage never fails intake. | `engines/analyzer.py`, `engines/openai_analyzer.py` |
| Work-type classification | Keyword-hit counting | `engines/analyzer.py` |
| Risk classification | Built-in keyword lists on ticket text + `fnmatch` globs on PR diff paths (default) **or** an LLM pass with cited evidence (`risk.provider: llm`). Both halves are operator-tunable without forking: the diff-stage path globs via `policy.sensitive_path_globs`, and now the ticket-text keywords via `risk.extra_sensitive_keywords` (issue #31, area name → extra keywords merged **on top of** the built-in floor via `merge_sensitive_keywords`; area names validated against `SENSITIVE_AREA_KEYS` at load). Strictly additive — extras can only flag *more* areas, never fewer, so they only escalate risk (invariant #1); no `SensitiveAreas` schema / `ApprovalRole` / Rego change (the keyword→area→level/roles mapping stays Python-side). The heuristics are a hard floor: the LLM may only escalate (add areas, raise the level), never downgrade — enforced in the classifier before the policy gate, so the policy engine/Rego are untouched. Evidence lands in the `RiskAssessment` artifact and `risk.escalated` event metadata. LLM failures degrade to the floor, recorded in the artifact. | `engines/risk.py`, `engines/llm_risk.py` |
| Repo routing | Tiered: explicit ticket association (conf 90) > delivery-memory priors (capped 89) > catalog IDF-token / keyword scoring (lexical-capped 89, stale-capped 65) > legacy YAML keywords. Lexical, not semantic. With `context.provider: code`, the synced file tree becomes a scored field and the bundle carries `RepoCodeFacts` (test layout, CODEOWNERS, manifests) + candidate files + inferred test commands; reasons cite concrete paths. Same confidence tiers/caps — code evidence never adds a tier. | `engines/enrichment.py`, `engines/code_context.py`, `catalog/`, `memory/priors.py` |
| Planning | Template rendering — steps are "Satisfy acceptance criterion: X" (default) **or** an LLM pass (`planner.provider: llm`) that turns the code-aware context (candidate files, `RepoCodeFacts`) into file-level steps, test locations, verify commands and `expected_files_or_areas`. The LLM is only consulted for runs the template planner would dispatch (ready + confident repo); identity/scope stay deterministic and the `agent_instructions` guardrail block is rendered by Foundry, never the model. LLM failures degrade to the template plan, recorded in the plan's `open_questions`. | `engines/planner.py`, `engines/llm_planner.py` |
| Delivery memory | Outcomes derived purely from the audit trail; Beta-smoothed routing priors (min 3 samples, >50% success); per-provider agent scorecards (which agent ships, by work type/repo); a scorecard-backed `recommend_provider` that turns the cards into one explainable provider pick for a work-type/repo (same priors guard rails: min-sample floor, >50%-merged gate, candidate allow-list); per-provider scorecard *trends* (`agent_scorecard_trends`) bucketing the same merge rate/retries/cost by day/week on one shared, zero-filled time axis so a card's direction of travel is visible; **learned dispatch has landed** (`agent.provider: auto`, issue #33): the orchestrator now *acts* on `recommend_provider` at first dispatch — see the `agents/` and `orchestrator.py` rows — so the scorecards are no longer reporting-only; ROI metrics API | `memory/` |

The deterministic heuristics are the no-key/offline reference implementations and
the test baseline; they are intentionally conservative, not unfinished LLM calls.

OIDC is opt-in (`auth.oidc` — `issuer`/`audience`/`jwks_uri`, all three required together or `Settings` load fails), additive (the static token is unchanged), and hardened (JWKS fetched through an injected transport seam so it tests offline; RS256-only algorithm allow-list shuts off `alg:none`/HS-confusion; `iss`/`aud`/`exp` required, bounded clock-skew leeway; unknown/missing `kid` refused). `pyjwt[crypto]` is the lazily-imported optional `[oidc]` extra — the default no-OIDC path never imports it; a deployment that sets OIDC without the extra fails loud at startup. **OIDC approver binding + IdP-group → role mapping (#34):** on the REST `POST /runs/{id}/approval` path, when the request authenticated via OIDC the approver *identity* is the **verified** subject claim (`auth.oidc.subject_claim`, default `email`, fallback `sub`), **not** the request body — a body `user` disagreeing with the verified subject is refused (403) — and the approver's roles are the committed `approvers[identity]` grant **unioned** with the roles the verified group claim (`auth.oidc.group_claim`, default `groups`) maps to via the committed `auth.oidc.group_role_map`. The session cookie is signed by `api/sessions.py` (`SessionSigner`, HMAC-SHA256, stdlib only — integrity + expiry, *not* encryption) and authenticates the dashboard's **read** calls (`_require_api_token(..., allow_session=True)`); it is deliberately **rejected on the approval endpoint** (`allow_session=False`) so an auto-sent cookie can't be tricked (CSRF) into driving an approval — approvals still need a bearer/webhook. No `refresh_token` is stored (the signed-cookie/no-server-store design holds); additive, opt-in, fail-closed (cap must be `> 0`, `>= session_ttl_seconds`, and requires the browser-login config), default-off ⇒ the prior fixed-TTL cookie. ## Hard invariants — do not route around these

1. **Never weaken a gate.** PRs that loosen a policy rule, approval requirement,
   or audit write don't merge. Change rules explicitly, with tests, or not at all.
2. **Python policy engine and `foundry.rego` change together**, with tests in
   `tests/test_policy_engine.py` and `foundry_test.rego`. Lock-step is
   machine-verified: shared vectors in `tests/data/policy_vectors.json` run
   through `LocalPolicyEngine` (`tests/test_policy_parity.py`) **and** through
   `opa eval` (`scripts/policy_parity.py`, in the OPA CI job) against the same
   expected decisions — add a vector when you add a rule, on both sides.
3. **No network in core tests.** `pytest` must pass offline with zero API keys.
   New external calls need a transport seam + a fake.
4. **The policy gate is default-deny**; an unrecognised action is refused.
5. **Approver roles live in committed YAML / config**, never in request payloads.
6. **Secrets never reach agent prompts** — job inputs pass the leak scan in
   `agents/provider.py`.
7. **Forbidden-path blocks are never retried.** Blocked stays blocked — and that
   must hold under concurrency too: every run state-transition reads the run row
   with `_require_run(..., lock=True)` (`SELECT … FOR UPDATE`, no-op on SQLite,
   real on Postgres) and re-verifies status under that lock before writing. A
   (re)dispatch that finds the run already terminal (e.g. a human `stop` won the
   race) records the launched job, cancels it, and never reverts the status. The
   `(run_id, sequence)` audit index is unique so a duplicate sequence fails loud.
8. New config goes in `foundry.example.yaml` with a comment; secrets are env vars.

## Known limitations (so you don't rediscover them)

- Context/routing is lexical (token overlap), not semantic — no AST or embeddings.
  The `code` provider reads file trees/CODEOWNERS/manifests, but `static`/`catalog`
  still read no code, and code facts only exist after a `--code-facts` sync.
- Plans are templated by default; `planner.provider: llm` adds a file-level
  planner that consumes the code-aware context (candidate files, code facts).
  It populates `expected_files_or_areas`, and the **plan-vs-diff drift check now
  consumes it**: a PR changing files outside *every* declared file/area escalates
  the run to `REVIEW_REQUIRED` (orchestrator-only, escalate-only, `category:
  plan_scope_drift`; toggle `policy.enforce_plan_scope`, default on but inert
  when the plan declares no expected areas — the template planner declares none).
  A *plan-aware* gate that reasons about whether the change satisfies the plan
  (beyond file scope) remains future work.
- Risk-from-ticket is keyword matching by default (risk-from-diff globs are
  precise); `risk.provider: llm` adds judgment + cited evidence, escalate-only.
  Both heuristic halves are operator-extensible without forking — the diff globs
  via `policy.sensitive_path_globs`, the ticket-text keywords via
  `risk.extra_sensitive_keywords` (#31, merged onto the built-in floor, strictly
  additive). Genuinely new, dynamically-named risk *categories* (beyond the seven
  fixed `SensitiveAreas`) remain open on #31, alongside live user-loadable OPA
  bundles.
- Single-tenant DB (no `org_id` / row-level isolation yet). API auth accepts
  OIDC JWTs as well as the static bearer token (#34), and the OIDC REST-approval
  path now **binds the approver identity to the verified token** and **maps IdP
  groups to approver roles** via committed config (`auth.oidc.group_role_map`) —
  so SSO group membership can grant approval authority, the group→role map stays
  committed config (invariant #5), and the gate's role requirements are
  unchanged. The dashboard's **browser-side OIDC login (SSO)** has now landed
  (`api/oidc_login.py`: authorization-code + PKCE, signed session cookie via
  `api/sessions.py`) — the session cookie authenticates the dashboard's reads
  but is rejected on the approval endpoint (CSRF-safe), so it never widens the
  human gate. **RP-Initiated (federated) logout** has landed too
  (`auth.oidc.end_session_endpoint` (+ optional `post_logout_redirect_uri`),
  #34): `/dashboard/logout` ends the IdP SSO session, not just the local cookie,
  so a shared workstation can't silently re-authenticate — additive, opt-in,
  default-off. **OIDC session refresh** has landed too as **sliding sessions**
  (`auth.oidc.session_max_lifetime_seconds`, #34): set it and `session_ttl_seconds`
  becomes the *idle* timeout while this is the absolute cap — each authenticated
  request slides the session cookie forward (an `app.py` middleware calling
  `OidcLogin.renew_session`, which carries the original login `iat` and clamps the
  new expiry to `iat + max_lifetime`), so an active operator isn't logged out
  mid-task while an abandoned session still expires and a real re-login is forced
  periodically; signed-cookie/no-server-store design preserved (no `refresh_token`
  stored), additive/opt-in, default-off = the prior fixed-TTL cookie. Still no
  SCIM and no `org_id`/row-level isolation. Coarse per-client rate limiting fronts
  the webhook + API surfaces.
- Temporal driver now CI-proven against a **real** server (issue #37); the
  inline driver remains the production path. The known workflow holes
  (wait-timeout handling, decision-verb validation, intake idempotency,
  single-shot PR observation) are fixed and CI-tested under the time-skipping
  environment, activity retry/timeout policies are now explicit per-activity
  (`activity_options.py`), a compensation activity now auto-fails a run
  (`fail_run`) when an activity exhausts its retries so it can't strand active,
  and the workflow now also runs end-to-end against the real
  `temporalio/auto-setup` server in the `temporal` CI job (the docker-compose
  Temporal profile) — so durability is proven against the production server
  binary, not just the in-memory time-skipping harness. (Multi-day *wait*
  behaviour under wall-clock time stays covered by the time-skipping job, since
  a real server doesn't skip time.) The fleet
  dashboard now ages all three in-flight parked states — the approval queue
  (`/metrics/approvals`), the agent-execution queue (`/metrics/executions`), and
  the PR review queue (`/metrics/reviews`, "PRs sitting unreviewed for N hours",
  `dashboard.review_sla_seconds`) — each read-only with its own optional SLA. The
  execution queue additionally carries each in-flight run's **agent spend so far**
  (`cost_usd`) with its own optional **cost SLA** (`dashboard.execution_cost_sla_usd`,
  `cost_breaches`/`costliest_execution_usd` on the strip) — the "which agent is the
  expensive one, and has it burned past `$N`?" signal, fired before the hard
  `policy.max_cost_per_run` cap; default unset = inert. The
  review queue additionally carries a "stale since last push" age per PR
  (`inactive_seconds`, dated from the latest `PR_OPENED`/`PR_UPDATED`) with its own
  `dashboard.review_stale_sla_seconds` knob (`reviews_stale`/`oldest_inactive_seconds`
  on the strip) — distinguishing an actively-pushed PR from an abandoned one,
  distinct from the open-age review SLA. Alongside those three *in-flight* queues
  the dashboard also has a **failure-side** triage feed (`/metrics/failures`,
  `failure_queue`): recently `BLOCKED`/`EXECUTION_FAILED` runs, newest-first,
  bounded to a recent window (`days`, default 7), each with how long ago it failed
  and why (the reason from the `RUN_BLOCKED`/`AGENT_FAILED` audit event) — the
  "what broke and needs a human" cut — plus its **aggregate** roll-up
  (`/metrics/failures/by-category`, `failures_by_category`): the same recent
  failures grouped by reason, most-frequent first, so the *systemic* blocker is
  visible at a glance, not buried in the feed's scroll. There is a **repo** axis
  too (`/metrics/failures/by-repo`, `failures_by_repo`): the same recent failures
  grouped by routed repo (the latest agent job's repo via `_run_repo_map`, since
  `FoundryRun` has no repo column; `(unrouted)` for runs that never dispatched) —
  the "is *one repo* the systemic blocker?" cut, the failure-side mirror of
  `delivery_by_repo`. There is a **work-type** axis too
  (`/metrics/failures/by-work-type`, `failures_by_work_type`): the same recent
  failures grouped by work type (derived per run from the latest `TICKET_ANALYSIS`
  artifact's `work_type` via `_run_work_type_map`, since `FoundryRun` has no
  work-type column; `(unclassified)` for never-classified runs) — the "do bugs
  fail while features ship?" cut, the failure-side mirror of
  `delivery_by_work_type`. The failure surface also
  has a **time** axis now (`/metrics/failures/trends`, `failure_trends`): the same
  recently-failed runs bucketed by *when they failed* onto one zero-filled axis
  (`day`/`week`), each period a count with a blocked/failed split — the
  *direction-of-travel* cut ("are we failing more than usual?") the recency feed
  and the point-in-time by-category roll-up can't show, what `delivery_trends` is
  to the delivery side. All three are read-only with an offline `foundry-memory`
  twin (`failures` / `failures-by-category` / `failures-by-repo` /
  `failures-by-work-type` / `failures-trends`); a blocked run
  stays blocked (invariant #7). The dashboard surfaces all of them in-page: the
  per-run failure feed, the by-category roll-up, the **by-repo roll-up**, the
  **by-work-type roll-up**, and the
  **failure-trend panel** (a by-week table from `GET /metrics/failures/trends`,
  bars splitting blocked vs execution-failed, scaled to the busiest week) — so for
  every failure cut the metrics derivation, endpoint, CLI twin **and** dashboard
  widget have all landed.
- One run targets one repo. The parent/child run model + epic rollup (#35) has
  landed (`epics.py`, `foundry_runs.parent_run_id`, `GET /runs/{id}/epic`), and
  the epic's full chain now exports as one cross-run evidence pack
  (`build_epic_evidence_pack`, `GET /runs/{id}/epic/evidence`), and the dashboard
  surfaces epics as a rolled-up board (`GET /epics`), and the *producer* —
  decomposing an epic ticket into independently-gated child plans — has landed
  (`engines/decomposition.py`, `orchestrator.intake_epic`), and the webhook/trigger
  intake path can now opt into running it automatically (`epics.auto_decompose`,
  default off, wired through `InlineDriver`), and LLM-assisted decomposition has
  landed (`engines/llm_decomposition.py`, `decomposition.provider: llm`) — it
  recovers prose-described epics over the deterministic floor (additive,
  grounded, degrade-to-floor), so the producer is no longer deterministic-only.
  Path-scoped policy has two slices now: per-repo forbidden-path
  globs (`policy.repo_forbidden_globs`) make the sticky forbidden-path block
  monorepo-aware (additive, no Rego), and per-repo **required approval roles**
  (`policy.repo_required_roles`, issue #31) let a repo demand specific approval
  roles regardless of risk (additive, lock-step Rego). The **N-of-M approval
  matrix** (the "two-person rule") has landed too (`policy.min_approvals` +
  per-repo `policy.repo_min_approvals`, issue #31): a run needs that many
  *distinct* human approvers before it leaves `WAITING_APPROVAL` for `APPROVED`
  (approvals accumulate, each its own audited sign-off, a duplicate from the same
  person refused). It is enforced in the orchestrator lifecycle — like the
  forbidden-path block, **no policy-engine/Rego rule**, so invariant #2 doesn't
  apply — and is strictly additive (default 1 = the historical single-approval
  path; a per-repo value only ever raises the floor via `max`, never lowers it).
  When the effective count is >1 the intake **approval prompt** (tracker comment +
  Slack message) surfaces it ("Approvers required: N distinct sign-offs"), so the
  first approver is told up front that one sign-off won't release the run rather
  than discovering it as a still-parked run; default 1 renders no such line. And
  once a *partial* sign-off lands, the run no longer goes silent: `approve()`
  fires a **mid-flow progress nudge** (`RunNotifier.approval_progress` +
  `format_approval_progress_comment`, issue #31) re-pinging the next approver on
  the tracker + chat ("N of M distinct sign-offs collected — K more required");
  presentation-only (no gate/audit change), and `min_approvals=1` never reaches
  the partial branch, so the single-approval path is byte-for-byte unchanged.
  A **starter policy library** has landed too (`policy/library/`,
  `foundry-policy` CLI, issue #31): vetted `baseline`/`soc2`/`change-management`/`pci-dss`
  presets a buyer copies into `foundry.yaml` — built only from existing knobs,
  copy-to-adopt (never auto-applied), so it adds no gate mechanism. The library
  is now also **checkable**, not just copyable: `foundry-policy check
  --against <preset-or-path>` (`compare_policy_strictness`) verifies a
  deployment's own config is at least as strict as a baseline on every gated
  knob and **exits non-zero when it is weaker** — a read-only compliance-CI gate
  that closes the adopt-then-drift gap, adding no policy mechanism. The dashboard
  runs that **same check continuously** (`GET /metrics/policy/check` + a
  compliance panel, issue #31): set `dashboard.policy_baseline` (a preset name or
  config path) and the live gate's control-by-control PASS/FAIL against that
  baseline is visible in-app — the always-on twin of the CLI gate, computed from
  the same `compare_policy_strictness` (shared `comparison_to_dict`), so drift
  below a committed baseline shows on the dashboard, not just in a one-shot CI
  run; default unset = no check, an unresolvable baseline fails loud at startup.
  **Per-*path*
  (sub-tree) approval roles** have landed too (`policy.path_required_roles`,
  issue #31/#35): a path glob → required approval roles, evaluated **diff-aware**
  in the orchestrator PR re-check (`_unapproved_path_roles`) — a diff touching a
  configured subtree whose role is not already covered by the run's approvers
  escalates the run to `REVIEW_REQUIRED` (audited `risk.escalated`, category
  `path_required_roles`). Like the forbidden-path block it is orchestrator-only
  and strictly additive (it can only ever *escalate* to human review, never
  release a run — invariant #1), so it adds **no policy-engine/Rego/gate rule**
  (invariant #2 doesn't apply); default empty = unchanged. **Change-freeze /
  maintenance windows** are the *time* axis of the same family
  (`policy.change_freeze_windows`, issue #31): recurring-weekly or absolute
  windows (`policy/freeze.py`) consulted before an autonomous re-dispatch in
  `_attempt_remediation` — during an active freeze the auto-retry is held and the
  run escalates to `REVIEW_REQUIRED` (audited `risk.escalated`, category
  `change_freeze`), so no code ships unattended during a release blackout. Same
  shape as the per-path rule: orchestrator-only, strictly additive (a freeze only
  ever holds work for a human, never releases it), surfaced in `explain` /
  `check` / the dashboard policy panels; default empty = unchanged; the initial
  human-approved dispatch is not gated. **Live user-loadable** OPA bundles and
  **custom risk categories** remain the open items on the programmable-policy
  epic (#31).
